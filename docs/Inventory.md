# Inventory

Item-centric inventory over the material sim. Items are the resource; materials are one kind of item — a thin layer, not a parallel economy.

- **Item model** (`core::item`): `ItemStack { item, count }`; the compiled item table holds named items plus one auto-generated material item per material — except air (the empty cell) and the player's `FLESH` body material, which share item 0 (`none`); fire and burning variants do get items. Item generation encodes no survival rules — obtainability in normal play is entirely the dig gate's job (phase, hardness, tool, inventory). Icons, stack caps, and tool metadata are authored in `fallingsand_content` and reached at runtime through `content::item(id)` — no registry object. Recipes are shapeless, count-based, and compiled to concrete item ids in `content::RECIPES`.
- **Slots**: player = 30 (hotbar 0..10, main 10..30); the server holds the cursor-held stack.
- **Dig / place**: each action addresses one server-selected cell, bounded to authoritative reach so raw aim never sets the work volume. Aim rays stop at obstruction; a held movement key clears the blocking cell in that direction instead. The server replicates the target, validity reason, and progress, and idle-previews only a currently-valid target. Progress binds to target, material, and mining method (hands or a specific tool); switching resets it. Only solid and powder cells dig; each yields an item — burning fuel yields its base material — while liquids, gases, and player rasters are not diggable. A cell with no item is `Undiggable`; matter is never silently voided.
- **Tools**: mining tier is derived from material hardness (≤0.35, ≤1.0, ≤2.0, else) at content-compile time. Bare hands are tier 0 at 0.55× speed; wooden, stone, and iron pickaxes are tiers 1/2/3 at 2×/3.2×/4.8× speed. A full inventory refuses the dig before matter is removed. Rigid-body cells use the ordinary body-damage queue.
- **Creative matter**: creative dig and place are explicit creator-power source/sink causes. They bypass inventory and tier requirements, but still use public unflagged writes, one-cell ownership checks, body damage, and the same authoritative target.
- **Slot actions**: server-authoritative and intent-based — the client resolves keybinds to intents, no raw modifiers cross the wire: left/right click, quick-move, trash, craft (once or all; a craft whose output wouldn't fit is refused — inputs consume against a trial copy, so crafts that free their own space work), creative grab. The server re-validates everything. Inventory rides the `TickFrame`: full state on first frame, then per-slot diffs.
- **Trash**: the one sanctioned mass-deletion affordance (items never drop into the world). A single trash slot, invisible to insertion, counting, and crafting by construction: cursor non-empty → previous trash contents are destroyed and the cursor stack moves in; cursor empty → the trashed stack returns (recoverable until replaced). Destroying is a single deliberate gesture.
- **Client UI**: `E` toggles a full-screen overlay (grid + hotbar + trash; side panel = crafting / creative palette); drag & drop via the authoritative cursor, no prediction; the hotbar is always visible.
- **Persistence**: `PlayerRecord` stores per-slot `StackRecord { item, count }`, keying each stack by the item's name; an unknown name on load is dropped. Plus cursor, trash, and bounded chat/command history. No migrations.

The initial progression is wood → planks → sticks → wooden pickaxe → shallow coal/iron → coal-assisted ingots → stone and iron pickaxes. Processing is inventory crafting; a furnace is intentionally deferred until fuel, timed processing, UI, persistence, and suspend/resume can ship as one feature.
