# Client

Display/input/UI only — never drives game logic. Bevy is the IO engine; game state is plain Rust with no bevy machinery in it.

## Architecture

- **`game/` — the core.** One plain `ClientGame` struct owns everything, with a `Flow` enum (`Menu | InGame`); `InGame` owns all session state, and dropping it (leaving to menu) is the entire cleanup path. At most one client-owned `GamePanel` (`Inventory | Chat | GameMenu`) is open. Death, revive, and connection screens derive from authoritative state instead of becoming stored panels. Session loss keeps `InGame` alive: replicated state clears, transient panels close, and the supervisor redials.
- **One update entry point.** `ClientGame::update(&mut self, io: &mut IoFrame)` runs fixed stages: UI events → input → net (demultiplex each message once) → flush fixed-cadence `InputFrame`s → timers. Opening the Game Menu or becoming incapacitated schedules one immediate neutral frame in the flush stage so held controls cannot leak across the transition. No bevy messages, states, change detection, or `FixedUpdate` in client logic.
- **`view/` — the bevy shell.** `Game(ClientGame)` is a single resource; one driver system builds the `IoFrame`, calls `update`, executes returned `Effect`s; every other system reconciles bevy entities/assets/UI from core state.
- **Update triggering.** Cheap values recompute every frame behind if-different guards; expensive updates gate on `Changes`, a frame-scoped signal struct. Spawns bake correct initial values — no first-frame patch.
- **Entity policy.** The core owns zero `Entity` ids and zero `Handle`s; view modules must render any core state from scratch — this is what makes reconnect and quit-to-menu clean.
- **Player projection.** One `PlayerId -> RemotePlayer` avatar directory is the source for every public pose, including the local camera and light. Private `SelfState` contains profile context, `SelfLife`, and a coarse camera `anchor`; only `SelfLife::Alive` carries health, air, and interaction. It never duplicates the alive pose: the camera follows the avatar while alive and falls back to `anchor` while entering, dead, or reviving, so it stays placed through respawns and reconnects instead of resetting to the origin.

## Rendering

- **Chunks**: one GPU texture per chunk, dirty sub-rect uploads; material + shade resolve to color via a palette texture in the shader. Players have no sprite — their flesh cells render as world cells (the shade pattern is the character art); nametags and camera derive from the replicated pose, cell-snapped, the camera's exponential smoothing keeping whole-cell steps readable.
- **Particles**: dig spray is server-authoritative. The client never decides when or where particles spawn; it drains `ParticleSpawn` events from each frame, spawns a short-lived sprite per event, and integrates velocity, gravity, and fade locally. Leaving to menu or session loss clears the queue and live particles.
- **Pixel-perfect layers**: every layer (world, sky, parallax) renders at native cell resolution (1 texel = 1 cell) into its own small HDR target, then upscales ×k onto a physical-pixel composite offset by its snap remainder — smooth camera, crisp k×k cells. Data-driven from one `LAYERS` table. Per-cell ambient multiplies at native texel centers (alpha-preserving); targets hold premultiplied color and composite materials must match. Bloom/tonemapping live on the composite.
- **Lighting**: ambient is per-cell and shaped by air. A normalized-blurred air-mask — material coverage inverted (open air = 1, water partial, solid = 0) — scaled by the sky's day/night level lights surfaces facing open space while cells sealed in rock stay dark at any hour, so buried ore is invisible until air reaches it. On top, emissive materials feed a glow field (every emissive cell contributes, tinted by its own material colour, softened for falloff, fire self-flickering) and player/burning point-lights — an exact position/radius/intensity list that warns rather than silently truncating — un-darken locally at night. The emissive pass renders over the viewport expanded by a blur-radius margin on every side, then a box downsample chain collapses it into one quarter-resolution light field — glow in color, air coverage in alpha — blurred once separably with precomputed gaussian weights, energy-preserving for glow and normalized for air; the composite bilinearly samples the expanded field at its margin offset, so light and darkness from just-off-screen cells resolve correctly instead of clamping at the frame edge. Every offscreen pass goes inactive outside gameplay. No stride, cap, or wall-clock throttle: each field follows world change as it happens.
- **Camera**: k = round(window_px/424) screen pixels per cell (~424×242 cells visible); `Ctrl`+scroll steps k by whole pixels between half and four times the base scale. Interest budgets are sized for the default zoom, so zooming out never grows server obligations.
- **Parallax**: client-only procedural layers keyed by ratio (0 = gameplay, 1 = sky): two mountain silhouettes — opaque black, made visible purely by airlight, so aerial perspective tracks day/dusk/moon brightness — and a dim cave wall behind dug-out caves.
- **Sky**: `Calendar::celestial()` in core is the single deterministic source — a native-2D "great wheel" cosmology where on-screen overlap equals eclipse math by construction. The 60-day seasonal orbit peaks near midsummer. Sun and moon are texture-free shader discs evaluated on their original logical pixel grids (48×48 for the sun and the moon's size-dependent grid), with one-logical-pixel coverage and stable disc-local detail. The sun keeps its orange-to-pale pixel palette and eclipse corona; the moon keeps a blue-grey pixel palette with procedural phase, craters, and umbra coloration. A parity fixture checks the simulator's shared constants. Explore with [skysim.html](skysim.html).

## UI & input

- **Debug overlay** (F3): world/player context left, performance/system right (fps, render-pass and server-tick timings — see [Tech.md](Tech.md)); F3+G rect visualizer, F3+N game mode.
- **Game Menu**: `Escape` opens it above death or connection status. It contains Resume, Settings, and one exit action. Embedded single player pauses and labels the exit `Save & Quit to Menu`; remote play keeps running and uses `Quit to Menu`.
- **Settings page**: reachable from the main menu and the Game Menu (its own opaque input context, `Escape` closes back to whatever is beneath). Toggles fullscreen, vsync, render mode (`pixel-perfect`/`smooth`/`retro` camera snapping), cursor mode (`smart`/`precise`), and UI scale (`75`/`100`/`125`/`150` percent, default `100`). The exact schema persists as JSON in `saves/settings.json` on native only — the wasm build resets settings each session; missing or invalid files use defaults. Settings reapply on boot through the `ApplyWindow` effect.
- **Inventory UI**: see [Inventory.md](Inventory.md) — `E` overlay, server-authoritative drag & drop, always-visible hotbar; world input is suppressed while an overlay is open.
- **Input and modal priority**: `ClientGame` derives one context stack from flow, authenticated-session readiness, `SelfLife`, the active panel, and Settings. Priority is Gameplay < Inventory/Chat < Incapacitated < Unavailable < Game Menu < Settings; initial Connecting replaces Gameplay. `game/input/` maps each opaque context through a declarative binding table over `RawInput`, with chords beating plain presses. The Bevy shell only snapshots keys and pointer state. Named UI depths mirror the same visual order: gameplay panels < death < connection status < Game Menu < Settings < debug.
