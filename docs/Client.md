# Client

Display/input/UI only â€” it never drives game logic. Bevy is the IO engine (winit window/input, `Material2d`+WGSL rendering, RenderApp texture streaming, bevy_ui widget tree, web bundling); game state is plain Rust with no bevy machinery in it.

## Architecture

- **`game/` â€” the core.** One plain `ClientGame` struct owns everything: registries, settings, menu state, input, view prefs, and a `Flow` enum (`Menu | InGame`). `InGame` owns all session state (net, phase `Connecting | Playing`, `paused`, world view, players, inventory, chat, clock, debug) â€” leaving to the menu drops it, and that drop is the entire cleanup path (the embedded server joins on drop). Session loss keeps `InGame` alive: `on_session_lost` clears replicated state and the supervisor redials; the connection screen doubles as the reconnect overlay.
- **One update entry point.** `ClientGame::update(&mut self, io: &mut IoFrame)` runs fixed stages: apply `UiEvent`s â†’ resolve input (context machine + hotkeys) â†’ net update (poll the connection, demultiplex each message once into domain `apply` methods; `Connecting â†’ Playing` flips when the first tick frame applies after join) â†’ flush input (manual accumulator, one `InputFrame` per tick, edges latched so they survive aliasing) â†’ tick timers. No bevy messages, states, change detection, or `FixedUpdate` anywhere in client logic.
- **`view/` â€” the bevy shell.** `Game(ClientGame)` is a single resource. One driver system (`drive_game`) builds the `IoFrame` (dt, key/button state, wheel, cursor cell, UI-hover, chat text, collected `UiEvent`s), calls `update`, and executes returned `Effect`s (screenshot, window mode, quit). Every other system reads `Game` and reconciles bevy entities/assets/UI from core state. Ordering is one declaration in `main.rs`: `collect_ui_events â†’ drive_game`, then a cameraâ†’skyâ†’lightingâ†’parallaxâ†’nametags chain, then independent view systems.
- **Update-triggering rules.** Cheap values (bars, labels, cursor follow) recompute every frame behind if-different guards. Expensive updates (slot patching, craftable scan, chat rows, world rows, side panel) gate on `Changes` â€” a frame-scoped signal struct the core fills during `update` and every view system reads the same frame. Roots (menu, HUD, pause, overlay, connscreen) are existence-checked per frame against core state. Spawns bake correct initial values from the core, so a fresh widget never needs a first-frame patch.
- **Entity policy.** The core owns zero `Entity` ids and zero `Handle`s. View modules own id maps (`ChunkVisuals`, `NametagVisuals`) or marker components and must render any core state correctly from scratch â€” this is what makes reconnect and quit-to-menu clean.

## Rendering

- **Chunks**: one GPU texture per chunk, re-uploaded only for dirty sub-rects, one quad each. Material id + shade resolve to color via a palette texture in a fragment shader â€” no per-pixel CPU loop. Uploads flow through a main-world queue extracted into the render sub-app (extraction never reads `Game`). Players have no sprite: their flesh cells arrive in chunk deltas and render as world cells (the `flesh` shade pattern is the character art); nametags, burning flames, lights, and the camera target all derive from the replicated cell pose in the core, cell-snapped and uninterpolated â€” the camera's exponential smoothing is what keeps whole-cell steps readable.
- **Pixel-perfect layers**: every layer â€” world, sky, and each parallax layer (`Layer` enum) â€” renders at native cell resolution (1 texel = 1 cell) into its own small HDR target sized `ceil(window/k)+padding` (even), one native camera each; the world camera floor-snaps to the cell grid, the rest are screen-space and scroll via their shader's snapped-cam. The composite camera lives in physical-pixel space (1 unit = 1 px) and is nothing but upscale quads: each samples one layer target, scaled Ă—k and offset by that layer's rounded snap remainder Ă—k â€” the camera moves smoothly while every cell stays a crisp kĂ—k block. The whole set is data-driven from one `LAYERS` table (ratio, z, render layer, follow/lit/drift flags); targets are recreated (not resized in place) on any native-size change by `resize_targets`, and `rebind_targets` reattaches every layer material from the `LayerTargets` change signal. Night darkness + emissive lights are evaluated at native texel centers on the world quad (chunky, cell-locked falloff) as an alpha-preserving multiply â€” air stays transparent, so lights never brighten the sky. Bloom/tonemapping live on the composite; UI targets it via `IsDefaultUiCamera`. Cameras spawn once at startup and are never despawned; the view resets their transform/zoom in the menu. GPU device loss recovers via `RenderErrorHandler` instead of exiting. F3+R cycles a test-only mode: pixel-perfect / smooth (fractional offsets, the old shimmer) / retro (whole-cell steps, no offset).
- **Camera**: k = round(window_px/424) screen pixels per cell (~424Ă—242 cells visible). `Ctrl`+scroll steps k by whole pixels between k/2 and 2k (min 1) â€” every integer k is pixel-perfect. Interest budgets are sized for the default zoom, so zooming out never grows server obligations. Name tags render on the composite in screen pixels, pinned above heads; F3 debug lines draw at screen resolution (fine 1px lines) from the same worldâ†’pixel mapping.
- **Parallax**: client-only procedural layers keyed by ratio r (0 = gameplay, 1 = sky), each rendered into its own native-res target by a fragment shader snapped to that layer's grid, then upscaled and composited back-to-front with its own sub-pixel offset â€” the same native-target-then-pixel-perfect-upscale path as world and sky. Two horizon-anchored mountain silhouettes (deterministic hash noise, cosmetic seed, opaque bodies tinted from sky color + darkness â€” aerial perspective, never see-through â€” that sink out of view by parallax when flying high) and a dim rocky cave wall (r 0.15) behind dug-out caves, lit by the same chunky lights and fading out at the surface. The sky is the r=1 native layer, and sun/moon are placed on whole-cell positions with whole-cell sizes, so their pixel art stays texel-rigid and steps as one unit instead of shearing sub-pixel.
- **Sky**: `Calendar::celestial()` in core is the single deterministic source â€” a native-2D "great wheel" cosmology where the rendered plane *is* the physics: on-screen overlap equals eclipse math by construction. An elliptical sun track behind the flat world gives night (the world blocks the sun), day length (8.1â€“15.9 h), and noon height in one mechanism over the 60-day year (four seasons). The moon laps the sun, breathes in size and speed, and drifts radially between rails â€” the 2D analogue of inclination, setting each lap's miss distance. Solar eclipse = the moon shrouding the sun, occlusion as exact covered-area fraction (total/annular/partial fall out of geometry). Lunar eclipse = the **Shade**, an invisible crimson-glass disc on a mirrored outer track, up only when the sun is down; the moon caught behind it turns blood-red â€” implied, never drawn. Golden-ratio phase steps keep eclipses irregular (~1 visible per 7 days). `CelestialState` (positions, illumination, occlusion, star visibility, sidereal time, `light`) is f32 and **client-visual only** â€” server gameplay gates use integer calendar math (`season()`, `day_of_year()`). Sun/moon/stars are pixel-art PNGs (`assets/sky/`) at one texel per world cell; shaders add bloom, horizon redness, phase terminator, and eclipse blood tint. Stars fade on sun depression, wash under moonlight, and reveal when `light` < ~0.3; the starfield is its own parallax layer (`L_STAR`, ratio 1) anchored in world-cell space (1 texel = 1 cell, so zoom is a true zoom) centered on the celestial center, drifting on a slow automatic time-based scroll â€” the integer-cell part feeds the shader, the sub-cell remainder rides the composite upscale offset like any parallax layer, so it slides in single physical pixels (no rotation â€” rotating a Nearest-sampled pixel bitmap shimmers). ## UI & input

- **Debug overlay** (F3): Minecraft-style two columns â€” left world/player context, right performance/system. Fast-changing counters are averaged over ~1 s; server-side stats are embedded-single-player only. F3+G toggles the rect visualizer; F3+N switches game mode.
- **Inventory UI**: see [Inventory.md](Inventory.md) â€” `E` overlay, server-authoritative drag & drop, always-visible hotbar. World input is suppressed while the overlay or chat is open.
- **Input**: `game/input.rs` owns every raw key/mouse read â€” one hardcoded keymap, one input context (chat > paused > inventory > gameplay), per-frame sampling that latches held state, queues actions, and blocks pointer presses starting over UI, flushed as one `InputFrame` per fixed tick so edges survive fixed-step aliasing. Local hotkeys mutate the core directly; buttons and slot clicks arrive as `UiEvent`s collected from the widget tree.
- **Keys**: WASD/arrows + Space move/jump, S crouch, double-tap Space creative flight, LMB/RMB dig/place, digits/scroll hotbar, `[`/`]` brush, E inventory, Enter chat, Esc back, F2 screenshot, F11 fullscreen, F3 overlay.
